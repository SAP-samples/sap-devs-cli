Three architecture approaches

How should the binary relate to the knowledge content? This is the most consequential architectural decision.

A

Embedded Content

Content is compiled into the binary via Go's embed package. One file, zero dependencies, instant startup.

Pros

  • Simplest possible distribution
  • Works 100% offline forever
  • No sync needed

Cons

  • Content updates require a new binary release
  • No company/user layer possible at runtime
  • Binary grows with every content addition
B

Thin Binary + Git Content Layers ★ Recommended

Binary is a lightweight engine. Content lives in git repos, synced to a local cache (~/.sap-devs/cache/). Three layers: official → company → user.

Pros

  • Content updates independent of binary updates
  • Company/team layer is just a second git repo URL in config
  • User overrides are just local files
  • Full offline support after first sync
  • Community PRs go to content, not code

Cons

  • Requires initial sync (or bundled seed content)
  • Slightly more complexity than A
C

Plugin Registry

Binary + an npm/OCI-style registry of installable packs. sap-devs pack install cap, sap-devs pack install abap. Packs are versioned independently.

Pros

  • Maximum granularity — install only what you need
  • Third parties can publish packs
  • Independent versioning per domain

Cons

  • Needs registry infrastructure to maintain
  • More complex install UX
  • Overkill for v1 — git already handles this

B is recommended: the git-layer model maps exactly to the official → company → user override structure you described, keeps the binary small, and lets content evolve without shipping new binaries. C's registry model can be added later if the ecosystem grows to need it.
